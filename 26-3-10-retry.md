# 什么情况下需要重试？
主要原则：只有可以恢复的情况下，才需要重试。
## 重试的场景有哪些？
 服务启动时，连接数据库， 连接缓存redis， 连接消息中间件 mq， 检查是否连通云存储。
 服务运行过程中 如果中间件连接异常，也需要重试
 第三方api调用的场景
 webhook机制
 mq消息的处理方案中重试
## 重试需要考虑的问题
 ### 重试的业务逻辑的要求 (api，rpc)
 - 被重复执行的轻量逻辑 支持幂等，以及 高并发
 - 被重复执行的逻辑如果比较重，费资源，费时间，费钱，最好支持不重复执行已经做过的环节
 ### 重试的策略
 - 基础中间件的重试策略：反复重试，直到成功；
 - api的重试策略：指数 退避策略，并且有上限。 达到上限仍然失败，就打印关键日志，或者记录到指定数据库表中，便于手动修复。
 - 定义清楚 成功的情况，明确哪些情况下失败 需要重试。
## 重试的异常有哪些？
网络波动导致 api调用异常
api 语义调整或者程序异常
对方服务异常
微服务中 上下游服务异常，接口异常；
